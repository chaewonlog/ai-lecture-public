## 공통 메타 패턴 10가지

### 1. 한 줄 목적 정의 (Frontmatter)

모든 스킬이 `description` 한 줄로 **"뭐 하는 스킬"인지 10초 안에** 파악 가능.

```yaml
description: 인사이트로그 뉴스레터 제목+부제 생성.
             조회수 데이터 기반 패턴.
             USE WHEN 뉴스레터 제목, 제목 후보, 타이틀, headline.
```

→ **언제 쓰는지 (USE WHEN) 명시** = AI가 정확하게 발동.

---

### 2. 트리거 조건 명시

```markdown
## 트리거
- `/today` 명령어로 실행
- 사용자가 "오늘 분석해줘", "오늘 시장 어때" 등으로 요청할 때
```

→ **"이 스킬이 언제 켜지는가"** 정의. 애매하면 안 쓰임.

---

### 3. 입력/참조 정의

```markdown
## 참조 파일
- 가이드라인: `/Users/kim-yewon/.../최종_투자_가이드라인.md`
- 포트폴리오: `/Users/kim-yewon/.../포트폴리오_현황.md`
```

→ **데이터 소스를 명시적으로**. AI가 알아서 찾게 두지 않음.

---

### 4. 데이터 기반 패턴 추출 (숫자로 증명)

```markdown
| 조회수 | 제목 | 패턴 |
|--------|------|------|
| 3,570 | 연말의 허무함은 '성과'가 없어서가... | A가 아니라 B (반전) |
| 2,160 | 릴리즈 노트...카피로 바꾸는 기술 | X를 Y로 바꾸는 (변환) |
```

→ 추상적 조언 ❌  
→ **실제 성과 + 실제 사례 + 패턴 이름** ⭕

---

### 5. 성공 vs 실패 대비 (반면교사)

```markdown
## 조회수 BOTTOM 5 (피해야 할 패턴)

| 조회수 | 제목 | 왜 안 먹혔나 |
|--------|------|-------------|
| 88 | IDE를 쓰기 시작한 진짜 이유 | 도구 이름이 일반적이라 호기심 부족 |
| 187 | 독립출판을 하는 삶 | 독자와의 접점 부족 |
```

→ **TOP만이 아니라 BOTTOM도**. 성공-실패 가르는 결정적 차이가 핵심.

---

### 6. 패턴에 "이름" 부여 (네이밍)

```markdown
### 공식 1: "A가 아니라 B" (반전형)
### 공식 2: "X를 Y로 바꾸는" (변환형)
### 공식 3: "1인칭 선언 + 약점/도발" (선언형)
### 공식 4: "도발적 질문" (질문형)
### 공식 5: "비개발자인 내가 ~한" (약점+도전형)
### 공식 6: "숫자 + 분석" (분석형)
```

→ 이름 없는 패턴 = 기억 안 남.  
→ **이름 붙은 공식 = 다음에도 바로 적용 가능.**

---

### 7. 함정 목록 (안티패턴)

```markdown
## 하지 말아야 할 것 (Gotchas)

- ❌ 제목이 30자 이상 넘어가면 임팩트 떨어짐
- ❌ 도구 이름만 제목에 넣지 않기
- ❌ "~하는 법"만으로 끝내지 않기
- ❌ 같은 패턴 3번 이상 반복 금지
```

→ **피해야 할 것을 명시** = 좋은 걸 알려주는 것만큼 중요.

---

### 8. 실행 프로세스 (번호 순서)

```markdown
## 실행 프로세스

1. 본문 또는 소재를 받으면, **핵심 메시지 1줄**을 먼저 뽑는다
2. 6가지 공식 중 **2~3개를 적용**하여 제목 후보 5개를 생성한다
3. 각 제목에 맞는 **부제**를 함께 생성한다
4. **Gotchas 체크**: 30자 초과? 추상적? 독자 접점 있는가?
5. 후보를 제시하고 채원님이 선택/수정
```

→ **step-by-step** 명시. "알아서 잘" 없음.

---

### 9. 출력 형식 템플릿

```markdown
### 출력 형식

​```
핵심 메시지: [한 줄]

제목 후보:

1. [제목] (공식: X)
   부제: [부제]
...
​```
```

→ **AI가 따를 정확한 템플릿 제공**. 매번 결과 형식이 흔들리지 않음.

---

### 10. 대비 구조로 설명 (Show, Don't Tell)

```markdown
5. **대비 구조로 설명**
   - "A는 ~합니다. B는 ~합니다" 식의 대비가 직관적입니다
   - 예: "ChatGPT는 방법만 알려줍니다. 에이전트는 '직접' 합니다"
   - 대안 비교는 **3열 표**로: `대안 | 한계 | 우리 차별점`
```

→ 설명 ❌ → **장면 보여주기 ⭕**  
→ 예시 먼저, 기술 설명은 뒤에.

## 글쓰기 톤 공통 규칙

| 원칙 | 적용 예 |
|---|---|
| **한 문장 = 하나의 정보** | 평균 20자 이내 |
| **접속사 금지** | "~하며, ~하고, ~함으로써" 쓰지 않음 |
| **경어체 통일** | "~합니다/~입니다" / 반말체 ("~함/~됨") 금지 |
| **단정적 종결** | "~일 수도 있습니다" 대신 "이미 되고 있습니다" |
| **구체 숫자** | "큰 기회" 대신 "일매출 1억" |
| **타이틀 연결** | 앞 타이틀 키워드를 다음 타이틀이 받음 |
| **기술 용어 배제** | "RAG" 대신 "AI가 자료 찾아서 답합니다" |

---

## 가치관 공통 규칙

1. **추상 대신 구체** — "시장이 성장" ❌ → "월 수백억 움직임" ⭕
2. **설명 대신 시연** — "이렇게 할 수 있습니다" ❌ → 시나리오 박스로 장면 보여주기 ⭕
3. **실적으로 증명** — "말로만이 아닙니다 — 이미 작동하고 있습니다"
4. **FOMO / 욕망 자극** — "이미 남이 돈 벌고 있다" 가 "지금 아니면 늦는다"보다 강력
5. **쉬운 말 먼저, 기술명은 뒤** — 개념 설명 → 이름 붙이기

## 체크리스트 (새 스킬/프롬프트 만들 때)

프롬프트나 스킬을 설계할 때 이 체크리스트를 통과하는지 검증.

### 구조 (7개)
- [ ] 한 줄 목적 정의 (`description`)
- [ ] 트리거 조건 명시 (USE WHEN)
- [ ] 입력/참조 정의
- [ ] 실행 프로세스 번호 순서
- [ ] 출력 형식 템플릿
- [ ] 실행 예시
- [ ] 체크리스트/Gotchas

### 콘텐츠 (6개)
- [ ] 데이터 기반 (숫자 + 사례)
- [ ] 성공 / 실패 대비
- [ ] 패턴에 이름 부여
- [ ] 표로 시각화
- [ ] 함정/안티패턴 명시
- [ ] Show, Don't Tell (시나리오 박스)

### 톤 (5개)
- [ ] 한 문장 = 하나의 정보
- [ ] 경어체 통일
- [ ] 불릿 위주 (줄글 최소화)
- [ ] 구체적 숫자·사례
- [ ] 단정적 종결

---

## 한 줄 요약

**데이터 기반 + 성공·실패 대비 + 패턴 네이밍 + 함정 목록 + 실행 프로세스 + 출력 형식**

이 6가지를 항상 포함하면, AI가 **뻔한 답 대신 실제로 쓸 수 있는 결과**를 뱉는다.

---

## 활용 예시

### 역설계 프롬프트에 적용
```
아래 [결과물 종류]의 성공·실패 사례를 분석해서 "공식북"을 만들어주세요.

[성공 사례 5~10개] (성과 지표 포함)
[실패 사례 3~5개] (반면교사)

출력 요구사항:
1. 공식 TOP 5 (각 공식에 기억나는 이름 부여)
2. 성공 vs 실패의 결정적 차이 3가지
3. 함정 5가지 (피해야 할 것)
4. 실행 프로세스 (내가 직접 쓸 때 순서)

표·번호 형태로 정리. 뻔한 분석 말고 제가 놓쳤을 패턴 위주로.
```

→ 이 프롬프트 하나가 **6가지 원칙을 모두 강제**. 그래서 결과 품질이 흔들리지 않는다.
